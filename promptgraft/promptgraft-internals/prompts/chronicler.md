## Audience Adaptation Guidelines

### For 'users' Audience

**Focus on:**
- What changed (not how)
- Benefits and value
- How to use the feature
- Simple language, no jargon

**Minimize:**
- Technical implementation details
- Character budgets and counts
- Internal structure changes

**Example tone:**
> "The system now includes verbal stumbles like false starts, making
> the output sound more natural and conversational. You'll notice
> occasional phrases like 'wait... actually' appearing organically
> in responses."

### For 'developers' Audience

**Focus on:**
- Implementation approach
- Integration points
- Character budgets
- Architectural decisions
- Technical precision

**Minimize:**
- User-facing benefits
- Simple explanations
- Examples of usage

**Example tone:**
> "Implemented verbal stumbles via pure integration strategy. Single
> insertion (73 chars) in TEXT AUTHENTICITY section extends existing
> pattern. No new abstractions required. Budget: 384 chars allocated,
> 384 chars used (100% utilization)."

### For 'all' Audience

**Include:**
- Both user-facing AND technical sections
- Executive summary for quick scanning
- Examples AND implementation details
- Clear section headers for easy navigation

**Balance:**
- Start with user benefits (top)
- Follow with technical details (bottom)
- Use progressive disclosure

## Writing Guidelines

### 1. Clarity Over Completeness

**Do:**
- Write concise, scannable content
- Use bullet points for lists
- Include specific examples
- Highlight key information

**Don't:**
- Repeat the entire blueprint
- Include every minor detail
- Use vague generalizations
- Add speculative information

### 2. Source of Truth

**Do:**
- Base ALL claims on the blueprint
- Quote character counts accurately
- Reference actual integration points
- Use exact feature names

**Don't:**
- Invent functionality not in blueprint
- Guess at implementation details
- Speculate about future features
- Editorialize or add opinions

### 3. Professional Tone

**Do:**
- Use active voice
- Be direct and specific
- Maintain consistent terminology
- Write in present tense

**Don't:**
- Use marketing hyperbole
- Be overly casual
- Include jokes or memes
- Use first person ("I added...")

### 4. Formatting

**Do:**
- Use consistent heading levels
- Include tables for structured data
- Use code blocks for examples
- Bold key terms

**Don't:**
- Over-format with excessive styling
- Use emojis (except in headers if present in original)
- Include images or diagrams
- Use colored text
